# 01. Research Foundation

## 1. Project Context

Modern machine learning depends on fast tensor operations. Many of these operations run on GPUs, where performance is determined not only by the mathematical operation but by how well the implementation uses parallel hardware, memory layout, indexing, and compiler behavior.

GPU kernels are the small programs that execute these operations. A good kernel can make an operation significantly faster; a bad one may fail to compile, access memory incorrectly, produce numerically wrong results, or run slower than existing compiler-generated baselines. Large language models can generate code in many programming languages, but GPU kernel generation is a more fragile task than ordinary application code. A generated Triton kernel must satisfy several constraints at once: it must be syntactically valid, compile successfully, match the PyTorch reference numerically, and ideally outperform or approach TorchInductor, PyTorch's compiler backend. This project studies whether task-specific adaptation can improve a local code model's ability to generate Triton kernels from PyTorch reference functions.

## 2. Research Problem

General-purpose (and specialized) code models are not reliably adapted to one-shot Triton kernel generation. Even when a model can write plausible Python or PyTorch, Triton requires specialized knowledge about GPU execution, pointer arithmetic, masks, launch grids, block sizes, dtype handling, and supported `triton.language` operations.

The research problem is therefore not simply that GPU programming is difficult. The more specific problem is that current code models may lack enough task-specific alignment with Triton kernel constraints to generate correct and performant kernels without retries, examples, or human correction.

The central gap is empirical: we need to measure whether adapting a code model to this task improves its one-shot generation ability compared with the same model in its vanilla state.

## 3. Why the Current State Is Insufficient

Triton is a specialized and relatively newer programming stack compared with broadly represented languages and frameworks. A model trained mostly on general code may know the surface form of Triton but still fail on the details that matter in execution.

Common failure modes include: (Is there a way to not throw all of them out there? I know they exist but throwing them to my face doesn't really says anything, maybe we could just mention a few and others.. Is this allowed in scientific papers?)

- invalid Triton API usage,
- incorrect kernel launch signatures,
- missing or misplaced `tl.constexpr` parameters,
- shape and indexing mistakes,
- dtype mismatches,
- unsupported operations in `triton.language`,
- kernels that compile but produce wrong numerical results,
- kernels that are correct but slower than compiler-generated baselines.

This matters because correctness and performance cannot be assumed from code that merely looks plausible. In GPU kernel generation, a model's output must be tested against executable references and benchmarked under a fixed protocol. 

The vanilla model's current baseline confirms that this is a meaningful problem. Out of the locked 437-row held-out evaluation set, `qwen2.5-coder:14b` produced only 39 compiled-and-correct one-shot kernels, and only 18 were faster than TorchInductor. This creates a clear baseline for testing whether SFT and DPO improve the model.

## 4. Core Research Question

Does task-specific adaptation improve one-shot Triton kernel generation over a vanilla code model?

In this project, task-specific adaptation means first supervised fine-tuning and then Direct Preference Optimization. The baseline model is `qwen2.5-coder:14b`; the SFT model is `qwen2.5-coder:14b-tried`; the DPO model label is still to be determined.

## 5. Supporting Research Questions

The core research question is intentionally focused, but several supporting questions help explain what we want to learn from the experiment.

RQ1. What failure modes dominate vanilla `qwen2.5-coder:14b` on one-shot Triton kernel generation?

RQ2. Does supervised fine-tuning improve the rate of generated kernels that compile and pass correctness checks?

RQ3. Does DPO further improve generation quality beyond supervised fine-tuning?

RQ4. Can this adaptation strategy provide evidence for improving other specialized code-generation tasks where correctness and performance both matter?

These questions are ordered from diagnosis to intervention. First, we characterize the baseline. Then we test whether SFT improves it. Finally, we test whether preference optimization adds value beyond SFT.

## 6. Working Hypothesis

Task-specific adaptation will improve one-shot Triton kernel generation compared with the vanilla model.

More specifically, we expect `qwen2.5-coder:14b-tried` to produce a higher rate of kernels that compile and pass correctness checks than `qwen2.5-coder:14b` on the same locked held-out evaluation set. We also expect DPO to further improve the model by favoring outputs that better satisfy the correctness and performance criteria.

This hypothesis is intentionally falsifiable. If the adapted models do not improve over the vanilla baseline under the fixed evaluation protocol, then the hypothesis is not supported.

## 7. Falsifiability and Success Criteria

The project must be evaluated through measurable outcomes, not through subjective impressions of generated code quality.

The hypothesis would be supported if the adapted model shows improvement over the vanilla model on paired evaluation rows. Important evidence includes:

- increased compile success rate,
- increased correctness pass rate,
- increased rate of kernels faster than TorchInductor,
- reduced frequency of dominant baseline failure modes.

The hypothesis would not be supported if SFT and DPO fail to improve these metrics, or if they improve one metric while harming correctness or performance in a way that weakens the overall result.

Negative results are still scientifically valuable. If task-specific adaptation does not improve this model, the failure profile can still reveal where general-purpose code models struggle and what kinds of supervision are insufficient for specialized GPU code generation.

## 8. Why This Is Research, Not Just Engineering

The project includes substantial engineering: data generation, preflight checks, an evaluation harness, model runs, and statistics tooling. However, the engineering is not the final claim. It is the experimental apparatus used to answer a research question.

This is research because it tests a falsifiable claim under controlled conditions. The same locked evaluation set is used across model conditions, the metrics are defined in advance, and the comparison is based on executable correctness and benchmark results rather than anecdotal examples.

The goal is not only to build a Triton-generation pipeline. The goal is to determine whether task-specific model adaptation measurably improves one-shot Triton kernel generation, and to document what kinds of failures remain.

## 9. Scope and Boundaries

This project focuses on one-shot eval generation. The model receives a PyTorch reference function, input shapes, and input dtypes, then generates one Triton implementation.

The held-out evaluation set contains 437 synthetic fusion tasks with fixed tiers: 103 easy, 217 medium, and 117 hard. These eval rows are not used as training examples.

The project compares model conditions:

- vanilla `qwen2.5-coder:14b`,
- SFT `qwen2.5-coder:14b-tried`,
- DPO model, label TBD.

The project does not claim to solve all GPU kernel generation. It does not guarantee optimal kernels, prove generalization to every Triton workload, or replace expert kernel engineers. Its scope is empirical: measure whether adaptation improves this model on this locked Triton generation benchmark.
